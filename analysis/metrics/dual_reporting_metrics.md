# Двойной отчёт метрик поиска пептидов: оригинал против исправленной (±3)

P/R/F1 используют метрику поиска пептидов с допуском ±3 по разрезам. **orig** — метрика
ровно как в `manuscript_metrics.get_counts_for_protein` (это значение в таблицах статьи; в
ней есть баг затенения переменной, занижающий recall — и ТОТ ЖЕ баг есть в исходном
DeepPeptide, поэтому числа остаются сопоставимыми с оригинальной работой). **corr** —
исправленный матчер ±3. Приводим оба, а не переписываем. Источник:
`analysis/metrics/src/corrected_metrics.py` → `analysis/metrics/corrected_metrics.csv`.
Полная сводная версия (с MCC/AUC и разбивкой по pep/propep) — в
`analysis/metrics/big_metrics_table.md`.

Столбцы `Δ` = corr − orig. Два запуска Таблицы 1
(`esm2_aho_transition_bias_sparse`, `esm2_bond_loss_soft`) не переинференсятся (классы
моделей утрачены) — corrected = N/A, сохраняем их опубликованные orig-значения.

**Почему здесь нет MCC/AUC:** баг живёт только в *сегментной* метрике поиска пептидов
(сопоставление начал/концов целых пептидов с допуском ±3). MCC и AUC считаются на уровне
**остатков** (позиция-за-позицией positive/negative) — это совсем другой путь кода,
который никогда не вызывает `get_counts_for_protein`, так что они **не затронуты багом** и
идентичны значениям в `analysis/metrics/canonical_metrics.md`. Поэтому здесь не
дублируются; эта таблица показывает только ту метрику, что действительно меняется
(F1/precision/recall).

Легенда: `F1 all` / `recall all` — F1 / recall поиска пептидов, объединённые по
пептидам+пропептидам; `recall pep Δ` / `recall propep Δ` — разрыв corr−orig по recall
отдельно для пептидов и пропептидов.

поэтому **ранжирование моделей не меняется**.
