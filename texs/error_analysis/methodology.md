# Методология: как и что считается во всех отчётах

Этот документ — единая точка истины по тому, как в проекте устроены данные, сплит,
все метрики и расчёт похожести пептидов. Все остальные отчёты (`analysis/**`,
`texs/error_analysis/**`) опираются на определения отсюда; если в отчёте написано
«recall», «±3», «похожесть к train» — смысл задаётся здесь.

---

## 1. Данные и разбиение train / valid / test

- **Датасеты.** `uniprot_2022` — исходный препроцессированный датасет (как в оригинальной
  статье DeepPeptide). `uniprot_2026` — более новая ревизия, **не** полностью
  препроцессированная (нет обработки фланкирующих мотивов). Все основные результаты
  считаются на `uniprot_2022`; запуск `uni2026_run_esm2` — на 2026.
- **Гомология-разделённый сплит.** Белки кластеризованы инструментом **GraphPart** по
  попарной идентичности (`needle --threshold 0.3`, то есть кластеры разделены так, что
  между ними нет пар с идентичностью выше 30%). Получается 5 кластеров.
- **Разбиение по кластерам:** train = {0, 1, 2}, valid = {3}, test = {4}
  (значения по умолчанию в `get_dataloaders`, `src/train_loop_crf.py`). Никакой конфиг
  это не переопределяет.
- **Homo-срез (Таблица 3).** Тот же test-кластер 4, но ограниченный белками из
  `protein_id_homo.txt` (только *Homo sapiens*).
- **Выбор лучшего чекпойнта.** На каждой эпохе считается (F1_pep + F1_propep)/2 на valid;
  сохраняется лучший по этой величине чекпойнт, и именно он оценивается на test.

**На что опираемся в выводах.** Поскольку сплит гомология-разделённый на уровне
**белков**, test измеряет обобщение на незнакомые белки. Отдельно мы проверяем, что это
переносится и на уровень **сегментов** пептидов (см. §4) — иначе «обобщение» могло бы
быть иллюзией из-за консервативных мотивов, протекающих через сплит.

## 2. Метрика поиска пептидов с допуском ±3 (P / R / F1)

Это **основная** метрика, по которой приводятся P/R/F1 во всех таблицах.

- **Что считается совпадением.** Истинный пептид (или пропептид) считается найденным,
  если у какого-то предсказания И начало, И конец попадают в **±3 остатка** от границ
  истинного сегмента. Перекрывающиеся истинные аннотации схлопываются в одну группу
  (достаточно найти любого члена группы). ±3 — это окно допуска, принятое во всём
  DeepPeptide.
- **Откуда P/R/F1.** Recall = найдено / всего истинных групп; precision = найдено /
  всего предсказаний; F1 — их гармоническое среднее. Считается отдельно для `peptides`,
  `propeptides` и их объединения `all`.
- **Реализация:** `src/utils/manuscript_metrics.py::get_counts_for_protein`.

### Старая vs новая метрика (важно для однозначности)

В оригинальной реализации `get_counts_for_protein` есть **баг затенения переменной**:
внутренний цикл `for idx, row in pred_df.iterrows()` переопределяет `idx`, поэтому флаг
«найдено» пишется в строку с меткой *предсказания*, а не *истины*; когда у белка
предсказаний больше, чем истинных сегментов, попадание уходит в «фантомную» строку,
которую `groupby` затем молча отбрасывает. В итоге recall **занижается на ~2–4 п.п.**

- **«Старая» метрика** = как в оригинальном DeepPeptide (с багом). Эти значения мы
  **сохраняем**, потому что тот же баг есть и в исходной статье → числа остаются
  сопоставимыми с ней.
- **«Новая» метрика** = исправленный матчер (`analysis/errors/src/error_analysis.py::match_protein`,
  проверен на ручных случаях). Recall выше на +0.024…+0.044 (пептиды) / +0.011…+0.023
  (пропептиды) — почти константный сдвиг, **ранжирование моделей не меняется**.
- Двойной отчёт (старая рядом с новой) для всех запусков:
  `analysis/metrics/src/corrected_metrics.py` → `corrected_metrics.csv`,
  сводно — `analysis/metrics/big_metrics_table.md`.

Деление «старая/новая» относится **только** к P/R/F1 — баг живёт в сегментном матчере.

## 3. Метрики уровня остатков: MCC и AUC

- Считаются **по остаткам** (а не по сегментам): каждый остаток — это бинарная метка
  «принадлежит пептиду/пропептиду или нет», и по всем остаткам теста считаются MCC
  (коэффициент корреляции Мэтьюса) и ROC-AUC.
- У них **нет** версий «старая/новая» — баг сегментного матчера их не касается, значение
  единственно.
- Источник — свежий **fp32**-инференс (`infer.py`). Важно: финальная оценка на test во
  время обучения идёт в fp32, поэтому инференс тоже принудительно fp32 (раньше была
  ошибка — инференс в bf16 AMP флипал пограничные предсказания у низкоувернных моделей).
- MCC/AUC приводятся только там, где свежий инференс воспроизводит train-time P/R/F1
  (drift = max|Δ P/R/F1| ≤ 0.015); иначе **N/A**. Это консервативная защита: если
  числа разъехались, residue-метрикам из того же прогона тоже не доверяем.
- Две модели невосстановимы для инференса в принципе (класс отсутствует в коде):
  `esm2_bond_loss_soft_l005_w5_tau15`, `esm2_aho_transition_bias_sparse_trainable_zero`
  → у них MCC/AUC всегда N/A, но старые P/R/F1 (train-time) сохранены.

## 4. Похожесть пептидов к train (peptide similarity)

Используется, чтобы проверить новизну held-out пептидов и связать её с recall, а также
как ключ для AHO-анализа (помогает ли словарь на похожих vs новых пептидах).

- **Что с чем сравнивается.** Из каждого сплита извлекаются сегменты пептидов и
  пропептидов (слитые координаты), дедуплицируются до уникальных последовательностей.
  Каждый уникальный held-out (valid/test) пептид выравнивается против **всех** train-пептидов
  **того же типа** (пептид с пептидами, пропептид с пропептидами).
- **Чем выравнивается.** EMBOSS **`needleall`** — глобальное выравнивание
  Нидлмана–Вунша, то самое семейство выравнивателей, что использует GraphPart (поэтому
  числа согласованы со сплитом).
- **Как считается идентичность.** `identity = matches / alignment_length` (дефолт
  `needle`, согласовано с GraphPart). Для каждого held-out пептида берётся **максимальная**
  идентичность к любому train-пептиду (`max_identity_to_train`).
- **Дополнительно coverage.** `coverage = matches / min(len_query, len_db)` —
  ловит случай «короткий фрагмент внутри длинного» (containment), который идентичность
  по длине выравнивания недооценивает.
- **Порог.** `is_similar_70 = identity ≥ 0.70` — порог «похож на train», предложенный
  научным руководителем; пептиды ниже считаются «новыми».
- **Реализация:** `analysis/similarity/src/peptide_similarity.py` →
  `analysis/similarity/peptide_similarity.csv` (одна строка на уникальный held-out
  пептид: seq, type, split, length, n_occurrences, organisms, `max_identity_to_train`,
  `coverage_at_best`, `best_train_seq`, `is_similar_70`).

**На что опираемся в выводах.** Когда мы говорим «held-out пептиды новые» — это значит,
что медианная `max_identity_to_train` ≈ 0.35 и лишь ~6% пептидов имеют ≥70%-идентичного
двойника в train. Когда говорим «recall падает там, где данных мало» — мы коррелируем
**среднюю по организму** `max_identity_to_train` с peptide recall этого организма
(Пирсон r ≈ 0.56). Это корреляция, не причинность; вторая ось — **обилие в train**
(организм с низкой похожестью, но многочисленный, всё равно хорошо распознаётся).

## 5. Общие принципы интерпретации (применимы ко всем отчётам)

1. **Что авторитетно.** Опубликованные P/R/F1 — это train-time `test_metrics.json`
   (fp32). Свежие MCC/AUC прикрепляются только при drift ≤ 0.015, иначе N/A.
2. **Один источник, один инференс.** Все «новые» метрики и residue-метрики берутся из
   детерминированного fp32-инференса (зафиксированы сиды; см. `src/utils/seeding.py`),
   так что числа воспроизводимы бит-в-бит.
3. **Шум одиночного запуска** ~±0.02 F1 (оценено по серии масштабирования данных).
   Поэтому различия меньше ~0.02 не считаем значимыми; выигрыш лучшей модели (+0.05 F1)
   — это ~2.5× шума.
4. **AUC по сегментным классам шумный** и калибровочно-чувствительный во всех запусках;
   как основную метрику качества используем F1 (сегменты) и MCC (остатки), AUC — справочно.
5. **N/A — это честно.** Ячейка N/A означает «нет надёжного числа», а не ноль. Строки
   без запуска в `runs/` в таблицы не попадают вовсе (а не как N/A).
