# Какие метрики репрезентативны в этой задаче и по какой отбирать модели

## 0. Дисбаланс классов (TEST, на уровне остатков)

| класс | доля позитивных остатков |
|---|---:|
| пептиды | 8.8% |
| пропептиды | 9.8% |
| любой (pep ∪ propep) | **18.6%** |
| фон (none) | **81.4%** |

(309 829 остатков на TEST; источник — `train_run_esm2/test_metrics_infer.json`.)
Дисбаланс **умеренный** на уровне остатков (~1:5), но на уровне **сегментов** он
резче: истинных пептидных групп ~1 тыс., а кандидатных предсказаний может быть
сопоставимо или больше, и «негативного класса сегментов» как такового нет.

## 1. Ключ к путанице: метрики живут на ДВУХ разных уровнях

- **Сегментный уровень** (с допуском ±w по разрезам): **Precision / Recall / F1**.
  Это и есть наша задача — *найти отрезок пептида с правильными границами*.
- **Поостаточный уровень** (метка на каждом остатке): **MCC, ROC-AUC, (AUPRC)**.
  Считаются по 309k остаткам, игнорируя понятие «отрезок» и допуск ±w.

Почти вся путаница (и наблюдение «AUC падает с ростом F1») — оттого, что это
**измерения разных вещей**, а не «противоречивые оценки одного и того же».

## 2. Каждая метрика: формула + интуиция здесь + пригодность для отбора

### Recall (±w), сегментный
`Recall = TP / (TP + FN)`, где TP — истинная группа пептидов, у которой нашлось
предсказание с началом И концом в пределах ±w.
**Интуиция:** доля реальных пептидов, которые мы поймали. **Не пропустить кандидата.**
Критичен, если задача — *скрининг/находка* (потом валидируют экспериментально).

### Precision (±w), сегментный
`Precision = TP / (TP + FP)`, FP — предсказанный сегмент, не совпавший ни с одной
истинной группой.
**Интуиция:** какая доля наших предсказаний — настоящие пептиды. **Не нагенерить
ложного.** Критична, если задача — *уверенная аннотация*.

### F1 (±w), сегментный
`F1 = 2·P·R / (P + R)` — гармоническое среднее.
**Интуиция:** баланс «не пропустить» и «не наврать». **Макро-F1 = (F1_pep + F1_propep)/2**
— ровно то, по чему обучение УЖЕ выбирает лучший чекпойнт на valid. Прямо отражает
задачу. → **Главная метрика отбора.**

### MCC (поостаточный)
`MCC = (TP·TN − FP·FN) / sqrt((TP+FP)(TP+FN)(TN+FP)(TN+FN))`, диапазон [−1, 1]
(0 = случайно).
**Интуиция:** «сбалансированная корреляция» предсказанных и истинных меток остатков —
учитывает все 4 клетки матрицы ошибок, поэтому **устойчив к дисбалансу** (в отличие от
accuracy, которая дала бы ~0.81 «бесплатно» за предсказание сплошного фона). У нас MCC
идёт **вместе** с F1 (победитель — лучший и по MCC 0.765, и по F1; AFT — худший по обоим).
**Минус:** это всё ещё уровень остатков — игнорирует допуск ±w и «отрезочную» природу.
→ **Вторичная проверка одной цифрой.**

### ROC-AUC (поостаточный)
`AUC = P(score(случайный позитив) > score(случайный негатив))` = площадь под кривой
TPR–FPR; диапазон [0, 1], 0.5 = случайно. **Threshold-free** (по всем порогам сразу).
**Интуиция:** насколько хорошо модель *ранжирует* остатки, безотносительно конкретного
порога и границ. **Почему здесь вводит в заблуждение:**
1. Меряет не то: ранжирование по всем порогам, а не нашу *операционную точку* (декод CRF)
   и не *границы* (±w). Модель может отлично ранжировать остатки, но после декода CRF
   выдавать сегменты с неверными границами/числом.
2. На дисбалансе ROC-AUC **оптимистичен**: доминируют многочисленные лёгкие негативы
   (81% фона), так что высокий AUC достижим без хорошей работы на редком позитиве.
3. **Инверсия с F1 — это by design.** boundary-голова победителя обостряет *декодируемые*
   границы (precision на операционной точке ↑), но делает ландшафт поостаточных скоров
   «острее»/хуже откалиброванным → гладкость ранжирования ↓ → AUC ↓. Поэтому
   `esmc_6b` (без головы) имеет AUC 0.766 но F1 0.579, а `esmc6b_boundary` — AUC 0.609 но
   F1 0.657. Это не «ухудшение», а смена того, что оптимизируется.
→ **НЕ использовать для отбора.** Годится только как *диагностика латентного сигнала*:
высокий AUC при низком F1 (ProstT5 0.783/0.509, AFT 0.732/0.382) = «эмбеддинг хорошо
ранжирует, но голова/порог не дотягивают» — сигнал, который лучшая голова могла бы
использовать.

### AUPRC / PR-AUC (поостаточный) — НЕ считаем сейчас, стоит добавить
Площадь под кривой Precision–Recall; базовый уровень = доля позитива (~0.19), а не 0.5.
**Интуиция:** как ROC-AUC, но смотрит на *редкий позитивный* класс и не «разбавляется»
лёгкими негативами. Для нашего дисбаланса информативнее ROC-AUC. → рекомендуется как
threshold-free диагностика **вместо** ROC-AUC.

## 3. Почему AUC анти-коррелирует с F1 (конкретные числа, TEST, all)

| модель | F1 (±3) | MCC | ROC-AUC |
|---|---:|---:|---:|
| ESM-C 6B + boundary/bond (победитель) | **0.657** | **0.765** | 0.609 |
| ESM2 (базовая) | 0.607 | 0.750 | 0.713 |
| ESM-C 6B (базовая) | 0.579 | 0.758 | 0.766 |
| ProstT5 | 0.509 | 0.719 | 0.783 |
| ESM2 + AdamW | 0.560 | 0.729 | **0.812** |
| AFT (all) | 0.382 | 0.550 | 0.732 |

F1 и MCC ранжируют модели согласованно; ROC-AUC — против. Это подтверждает: **AUC
измеряет другое** (ранжирование остатков), а не качество детекции сегментов.

## 4. Рекомендация по отбору моделей

1. **Основная метрика: macro-F1(±w) = mean(F1_pep, F1_propep)** на уровне сегментов
   (совпадает с критерием выбора чекпойнта в обучении). Использовать **исправленный**
   матчер ±w (см. `methodology.md`), но сравнивать модели можно и по старому — ранги те же.
2. **Вторичная (одна цифра, сбалансированная): MCC (поостаточный).** Идёт вместе с F1,
   служит проверкой «не переоптимизировали ли границы в ущерб остаткам».
3. **Диагностика (НЕ отбор): ROC-AUC** — индикатор латентного сигнала эмбеддинга; по
   возможности заменить/дополнить **AUPRC** (информативнее на дисбалансе).
4. **Окно ±w — это часть выбора метрики.** Свести модели по ±0/1/2/3 и (а) выбрать
   допуск под биологическую осмысленность границ, (б) проверить *устойчивость*: модель,
   держащая F1 при ±1, реально лучше по границам, чем та, что живёт только на ±3. Это
   запланированный sweep окон.
5. **Precision vs recall — под задачу.** Скрининг/находка новых пептидов → взвесить recall
   (Fβ, β>1). Уверенная аннотация → precision (β<1). По умолчанию — F1 (β=1).

**Короткий ответ:** отбирать по **macro-F1(±w)** (+ MCC как страховка), **AUC — только
диагностика, не критерий**; и зафиксировать окно ±w через sweep, а не брать ±3 по
инерции.
